# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run tauri dev` — run the full desktop app (Vite + Rust backend). This is the normal dev loop. **After adding a migration, quit and relaunch — hot reload won't re-run migrations.**
- `npm run dev` — Vite only on port 1420. Rarely useful in isolation; Tauri's `beforeDevCommand` already launches it.
- `npm run build` — Vite production build into `dist/` (consumed by `tauri build`).
- `npm run tauri build` — produce a distributable desktop bundle.
- `cargo check --manifest-path src-tauri/Cargo.toml` — fast syntax/type check for the Rust backend without a full build.
- `python3 scripts/seed_test_data.py` — seed the live SQLite DB with fake trades/plans against the first active account. Writes directly to `~/Library/Application Support/com.adamwickwire.futuresjournal/futures-journal.db`; the app must have been launched at least once so migrations have run.

There is no test suite and no linter configured.

## Architecture

**Stack:** Tauri 2 + vanilla JS (no framework, no bundled UI lib) + SQLite via `@tauri-apps/plugin-sql`. Vite serves `src/` with `root: "src"` and builds to `dist/`.

**Hash-based SPA router (`src/main.js`):** The app is a single `#app` mount with a sidebar shell and a `#page` slot. Routes are a flat ordered table — **exact paths must precede their `:param` siblings** because the matcher walks the list and stops at the first hit. Pages export `render(params)` that returns either an HTML string or `{ html, mount(pageEl) }` for post-insert wiring. Navigation happens via `hashchange`; `refreshPage()` re-renders only the page slot. Pages that attach window-level listeners (drag-drop, etc.) must call `registerPageCleanup(fn)` so they're torn down on navigation.

**Data layer (`src/lib/`):** All SQL goes through `db.js` (`query`/`exec` wrappers over a singleton `Database.load("sqlite:futures-journal.db")`). Each domain has its own module — `trades.js`, `plans.js`, `accounts.js`, `tags.js`, `instruments.js`, `settings.js`, `analytics.js`, `images.js`, `export.js`, `backup.js`, `ledger.js`, `risk.js`, and pure math in `calc.js`. Pages should call domain modules, not `db.js` directly.

**Pure math lives in `src/lib/calc.js`:** P&L, risk, R-multiple, planned RR, and shape validators for trades/plans. No DOM, no DB — import and unit-reason freely. The app is opinionated: **futures only, stops required, R-multiples are the primary metric, percentage P&L is deliberately absent** (meaningless for futures). Plans require both stop AND target (an entry without a target isn't a plan).

**Rust backend (`src-tauri/src/lib.rs`):** Thin. Registers the SQL plugin with an inline migrations vec (edit this when adding a migration — see below) and exposes eight custom commands: `save_image`, `delete_image` (sandboxed to `images/` dir via canonicalized path check), `write_text_file`, `read_text_file`, plus four backup commands (`list_backups`, `write_backup`, `read_backup`, `delete_backup`) all sandboxed to `<app_data>/backups/`. Everything else is the stock `tauri-plugin-sql` / `tauri-plugin-dialog` / `tauri-plugin-opener` surface.

**Migrations (`src-tauri/migrations/NNN_*.sql`):** Forward-only, numbered. Adding one requires **both** dropping the SQL file in the migrations dir **and** appending a `Migration { version, description, sql: include_str!(...), kind: MigrationKind::Up }` entry to the vec in `lib.rs::run()`. SQLite doesn't support modifying CHECK constraints in place — when you need to widen one (e.g. adding a new enum value to a column), do the canonical `CREATE <new>; INSERT SELECT; DROP; RENAME` table rebuild (see `009_transfers_and_tx_types.sql` for the pattern). The initial schema seeds the `instruments` table with common futures contracts (ES/MES/NQ/MNQ/CL/GC/ZB/etc.) — `point_value = tick_value / tick_size` is the invariant, and `is_micro` on each row is the classification the risk engine reads.

**Account model — two parallel columns:** `accounts.type` is the legacy funded/cash bucket that the rule engine (`risk.js`, `computeDrawdownFloor`, `computeHeadroom`) reads. `accounts.category` is the granular classification introduced in migration 008: `combine | sim_funded | live_funded | cash | bank`. **`category` is the source of truth; `type` is derived** from the `type` field on each `ACCOUNT_CATEGORIES` entry, so `createAccount` and `updateAccount` write both columns atomically and they can't drift. New code should read `category`; leave `type` alone unless you're touching the rule engine itself. Firm-specific rules (Apex, Topstep, Tradeify, etc.) are never hardcoded — they're modeled by combining generic mechanisms with per-account values. Bank-category accounts are ledger-only and must be filtered out of trade/plan pickers.

**Risk engine (`src/lib/risk.js` + drawdown math in `accounts.js`):** `assessDraft()` is the async entry point used by the trade form — it fetches everything needed (instrument, open trades, daily P&L, drawdown context), calls the pure `evaluateTradeRisk(ctx)` with everything pre-computed, and returns `{ blockers, warnings, computed }`. Blockers can be overridden by the user with a recorded reason. `computeDrawdownFloor(account, trades, transactions)` is the single source of truth for the drawdown floor — it walks a time-ordered event stream and handles three modes (`static`, `eod_trailing`, `intraday_trailing`), the `dd_lock_offset` trigger (lock when peak ≥ start + dd + offset, floor freezes at start + offset), and the `dd_lock_on_payout` trigger (any withdrawal/payout locks immediately). Mini and micro contract caps are checked independently via `account.max_minis` and `account.max_micros` using `instruments.is_micro` to classify each trade. Consistency rule (best day ≤ N% of total) is evaluated via `consistencyStatus()` in `analytics.js` and is display-only — no pre-trade blocking, because it's an end-of-day calculation. `loadAccountRiskContext(accounts)` is the batched loader (two queries total) that dashboards use to avoid N+1 when rendering headroom across many accounts.

**Real money ledger (`src/lib/ledger.js` + `src/pages/ledger.js`):** The app tracks two parallel ledgers. The sim ledger is combine / sim-funded / live-funded trading P&L — fake money used to qualify against firm rules. The real ledger is the user's actual net worth: cash brokerages, the personal bank account, prop firm payouts received, and all fees paid to firms. `realMoneyLedger(accounts, trades, transactions)` is a pure function that returns `{ events, curve, totals, running }` — it includes deposits/withdrawals on cash+bank accounts, payouts from funded accounts (the sim→real bridge), all fee/reset/activation transactions, and cash-account trade P&L. It deliberately excludes funded-account trade P&L (live-funded counts only when withdrawn). Transfers between two real accounts are marked `internal_transfer` and excluded from `totals.net` so bank↔brokerage movement doesn't inflate net worth. The `/ledger` page renders the curve, per-category totals, fee burn by account (via `paid_for_account_id`), and a chronological event table.

**Schema shape to keep in mind:** `trades` and `plans` both reference `accounts` and `instruments`; `plans.trade_id` links a plan to the trade that executed it. `trade_tags` and `plan_tags` are join tables (plan_tags added in migration 003). `trade_images` is polymorphic — rows carry either `trade_id` or `plan_id`. `transactions` tracks deposits/withdrawals/payouts/fees/resets/activations and transfer pairs — the `linked_tx_id` column pairs the two sides of a transfer (`transfer_out` on source, `transfer_in` on destination, created atomically by `createTransfer()` which rolls back the orphan half on failure). `paid_for_account_id` optionally tags a real-money fee (e.g. combine subscription paid from the bank) with the funded account it covers. `settings` is a simple KV table accessed via `lib/settings.js`. `recomputeBalance()` is the pure balance derivation — `account_size + Σ(signed tx) + Σ(closed trade P&L)` — and is called on every write path that could affect it.

**Privacy mode:** A UI preference persisted in `settings` and applied at bootstrap via `setPrivacyMode()` in `lib/format.js` before the first render — money values are masked when on. Respect this when adding any code that displays dollar amounts.

**Safety UI default — maximal visibility:** Dashboard Today panels and risk/safety views should show every active account regardless of which rule columns are set. The card adapts to whichever rules are configured per account; don't filter accounts out of safety UI just because a particular column is null. Archived accounts (e.g. failed combines) should be hidden by default from analytics and the active account picker, with an opt-in toggle to re-include them for historical review.
